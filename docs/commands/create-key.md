# create-key

Generate a per-repository encryption key for at-rest encryption and shared blob encryption.

## Usage

```bash
envi create-key
envi create-key --force
```

## Description

`create-key` generates a cryptographically random key and writes it to `envi.maml` at the repository root. Once the file exists, Envi:

- Encrypts captured env values at rest in `~/.envi/store/` (and in the optional GitHub backup)
- Uses the same key for `envi pack` / `envi unpack`, in preference to manifest-based key derivation

`envi.maml` is intended to be **committed to the repository** so collaborators with repo access can capture, restore, pack, and unpack without any further setup.

## Why a Key File?

A few things change once `envi.maml` exists:

- **Captured env values are encrypted at rest.** If your `~/.envi/store/` is backed up to GitHub via [`envi global github enable`](/commands/global#global-github-enable), only people with the source repo can decrypt the backup.
- **Shared blobs use a stable key.** The default for `envi pack` is to derive the encryption key from a manifest file (`package.json`, `Cargo.toml`, etc.), but that key changes whenever dependencies change — every `pnpm install` invalidates older blobs. A committed `envi.maml` survives dependency updates.
- **No secret to share.** Every collaborator who clones the repo automatically has the key.

## What It Does

1. **Finds repository root** - Same logic as `capture`/`restore`
2. **Refuses if already set** - If `envi.maml` already contains an `encryption_key`, the command exits unless `--force` is passed
3. **Generates 32 bytes of random data** as a base64url string
4. **Writes `envi.maml`** - Creates a new file with a header comment, or inserts the key into an existing `envi.maml` while preserving other content

## File Format

A freshly created `envi.maml` looks like:

```maml
{
  # Generated by `envi create-key`. This key encrypts env values
  # stored at rest in ~/.envi/store/ and is used by `envi pack` /
  # `envi unpack` to share env files securely.
  #
  # Commit envi.maml so collaborators can decrypt. Do NOT commit it to
  # a public repository — anyone with read access can decrypt your
  # env values.
  encryption_key: "u3qZKY9GgqOMcKQ9s51G81HjpIB4WuKkn9Zh1Qkon_0"
}
```

`envi.maml` is also where any future per-repo Envi configuration would live — `encryption_key` is just the first field.

## Examples

### First-time setup

```bash
$ envi create-key
◐ Finding repository root...
ℹ Repository root: /Users/you/projects/myapp
✔ Wrote encryption_key to envi.maml
ℹ
ℹ Next steps:
ℹ   1. Commit envi.maml so collaborators can decrypt.
ℹ      Do NOT add envi.maml to .gitignore — it must be tracked.
ℹ   2. Make sure this repository is private. Anyone with read access
ℹ      can decrypt env values captured with this key.
ℹ   3. Run `envi capture` to write encrypted env values.
```

### Refusing to overwrite

```bash
$ envi create-key
ℹ Repository root: /Users/you/projects/myapp

 ERROR  envi.maml already contains an encryption_key. Re-run with --force to replace it.

 WARN   Replacing the key will make any previously captured stores unreadable until re-captured.
```

### Forced rotation

```bash
$ envi create-key --force
ℹ Repository root: /Users/you/projects/myapp
✔ Wrote encryption_key to envi.maml
```

After rotating the key, run `envi capture` to re-encrypt the store with the new key. Any previously stored data encrypted with the old key cannot be restored.

## Security Considerations

::: warning Repo visibility matters
Anyone with read access to the source repository can decrypt env values captured with this key. **Only commit `envi.maml` to repositories that are private** (and that you trust the access list of).

If your project is or might become public, do not run `create-key`. Stick with `envi pack` using a custom secret for sharing, and skip at-rest encryption.
:::

::: tip Key rotation
The key is 32 bytes of cryptographically random data. There's no built-in incremental rotation — `--force` generates a brand-new key and any data captured under the old key becomes unreadable. After rotating, re-run `envi capture` from a checkout that has the original env files.
:::
